axci

智能分析代码变更与 Rust crate 依赖关系，
精确选择需要运行的测试目标，
避免在昂贵的物理板卡上全量重复测试。

核心痛点

🔧
物理板卡资源昂贵
Phytium、RK3568 板卡占用时间长，每次 push 全量跑浪费数小时
📦
Monorepo 依赖复杂
修改一个 crate 可能影响多个下游，难以手动判断测试范围
📝
文档改动无需测试
改 README.md 却触发全量 CI，占用排队资源
🔄
规则难以复用
每个项目各自维护一套 CI 逻辑，无法统一更新和维护

三层架构设计

01
规则层 configs/test-target-rules.json
描述「哪些路径 / 哪些 crate → 触发哪些测试目标」。外置 JSON，业务差异在此收敛，不改代码逻辑。组件仓库可覆盖默认规则。
↕
02
分析引擎层 axci-affected affected_crates.sh
① Rust 引擎（优先）：git diff + cargo metadata + BFS 依赖传播
② Bash 脚本（回退）：功能等价，依赖 jq，构建失败时自动降级
↕
03
编排层 .github/workflows/test.yml
CI入口：detect job 将引擎输出转为 GitHub Actions matrix，驱动并行测试 job（axvisor / starry）

分析引擎核心算法

四步决策流程

① git_changed_files()
git diff --name-only <base_ref>
回退：HEAD~1；失败则 skip_all
② compute_crates() — 前缀匹配
cargo metadata 获取 workspace 成员路径
对每个变更文件 → 最长路径前缀匹配 → changed_crates
③ BFS 反向依赖传播
构建 rev_deps: dep → [dependents]
从 changed_crates 出发广度优先扩散 → affected_crates
④ decide_output() — 目标决策
按优先级：skip → run_all → selection_rules →
crate_rules → crate_path_rules → 兜底全量

BFS 核心代码（简化）

// 构建反向依赖图
let mut rev_deps: HashMap<_, Vec<_>> = HashMap::new();
for node in resolve.nodes {
    for dep in node.deps {
        rev_deps.entry(dep.name)
            .or_default()
            .push(node.id.name.clone());
    }
}

// BFS 传播
let mut queue = changed_crates.clone();
while let Some(cur) = queue.pop_front() {
    if let Some(dependents) = rev_deps.get(&cur) {
        for d in dependents {
            if affected.insert(d.clone()) {
                queue.push_back(d.clone());
            }
        }
    }
}

decide_output() 决策树

changed_files 为空?
  └─ YES → skip_all = true
全是非代码文件?
  └─ YES → skip_all = true
命中 run_all_patterns?
  └─ YES → return all_targets
changed_crates ∩ run_all_crates ≠ ∅?
  └─ YES → return all_targets
逐条评估规则 → selected[]
  selection_rules  (路径匹配)
  crate_rules      (crate + BFS)
  crate_path_rules (crate + 路径联合)
selected 为空?
  └─ YES → 兜底: return all_targets
  └─ NO  → return sorted(selected)

CI 编排：test.yml detect job

detect job 步骤流程

1. Checkout component
调用者仓库的代码 → ./component/
2. Checkout axci-rules
axci 仓库（含引擎 + 规则）→ ./axci-rules/
3. 安装 Rust + 构建引擎
cargo build --release axci-affected
continue-on-error 失败后自动降级到 Bash
4. Build dynamic matrix（核心）
引擎输出 → 过滤 AXVISOR_ALL → 生成 matrix JSON
5. Write step summary
在 Actions 页面输出决策报告（selection_mode / auto_reason / targets）

detect job 输出变量

变量含义
axvisor_matrixJSON 数组，每条一个测试 job
skip_all是否跳过所有测试
selection_modeall / subset / auto-skip 等
auto_reason自动选择原因（调试用）
resolved_targets最终解析的目标 key 列表

axvisor/test.yml（主测试入口）

  push:    { branches: [master] }
  pull_request:
  workflow_dispatch:
    inputs:
      test_targets: { default: 'auto' }
      base_ref:     { default: 'origin/master' }
      runs_on:      { default: '["self-hosted","X64"]' }

jobs:
  test:
    uses: yoinspiration/axci/.github/workflows/test.yml
          @test/auto-target-regression
    with:
      crate_name:   "axvisor"
      test_targets: ${{ inputs || 'auto' }}
      base_ref:     ${{ event.before || 'origin/master' }}
      axci_ref:     "test/auto-target-regression"
      axvisor_repo: "yoinspiration/axvisor"
      axvisor_ref:  ${{ github.ref_name }}

axvisor job 每步详解

① Clean up workspace
git clean -ffdx + rm -rf .cargo
清除遗留 config，防止 rustflags 翻倍
② Checkout component → ./component/
被测组件（本次 PR 的代码）
③ Checkout axvisor → ./axvisor/
测试平台，axvisor_ref=github.ref_name，含子模块
④ Install Rust nightly + ostool
targets: aarch64-unknown-none-softfloat, x86_64-unknown-none
⑤ Resolve + Apply patch
cargo metadata 定位 crate 路径
在 axvisor/Cargo.toml 追加：
[patch.crates-io] axvisor = {path="…/component"}
⑥ Download images（QEMU only）
cargo xtask image download <vmimage_name>
更新 kernel_path / rootfs 路径
⑦ Run QEMU / Board tests
QEMU: cargo xtask qemu --build-config ... --qemu-config ... --vmconfigs ...
Board: cargo xtask uboot ... --bin-dir /home/runner/tftp

axplat-dyn 与链接脚本关系

axplat-dyn 的 build.rs 在编译时生成 axplat.x → OUT_DIR，提供给链接器。axplat.x 内部 INCLUDE "link.x"，而 link.x 由 somehal/someboot 提供。

// axplat-dyn/build.rs
fn main() {
    let ld = include_str!("link.ld");
    println!("cargo:rustc-link-search={}", out_dir);
    // axplat.x 内: INCLUDE "link.x"
    fs::write(out_dir.join("axplat.x"), ld).unwrap();
}

规则配置详解

规则层次

非代码文件（无条件跳过）
"non_code": {
  "dirs":  ["doc/", "docs/"],
  "exts":  [".md", ".txt", ".png", ...],
  "files": ["LICENSE", ".gitignore"]
}
全量触发模式（run_all_patterns）
[
  "Cargo.toml",       // workspace manifest
  "Cargo.lock",
  "rust-toolchain.toml",
  "xtask/*",           // build system
  "kernel/*"           // 整个 kernel
]
"run_all_exclude_patterns": [
  "kernel/src/hal/arch/*" // 但排除架构子目录
]
精确路径规则（selection_rules）
规则 ID匹配路径目标
aarch64_pathkernel/src/hal/arch/aarch64/*
configs/board/qemu-aarch64.toml		aarch64全部（5个）
x86_64_pathkernel/src/hal/arch/x86_64/*
platform/x86-qemu-q35/*		qemu-x86_64-nimbos
phytium_path*phytium*, *e2000*phytiumpi×2
rk3568_path*rk3568*, *rockchip*rk3568-pc×2

crate 级规则

全量触发 crate（run_all_crates）
只要这些 crate 出现在 changed_crates（直接修改），立即全量
axruntime axconfig
crate_rules（crate 精确匹配）
{
  "id": "crate_x86_platform",
  "crates": ["axplat-x86-qemu-q35"],
  "targets": ["axvisor-qemu-x86_64-nimbos"],
  "direct_only": false  // 用 affected_crates
}
crate_path_rules（crate + 路径联合）
必须同时满足：crate 命中 AND 文件路径匹配，顺序评估，首命中即停 
[
  { "id": "driver_phytium",
    "crates": ["driver"],
    "path_patterns": ["*phytium*", "*e2000*"],
    "targets": ["phytiumpi-arceos", "phytiumpi-linux"] },
  { "id": "driver_rockchip",
    "crates": ["driver"],
    "path_patterns": ["*rk3568*"],
    "targets": ["roc-rk3568-arceos", "roc-rk3568-linux"] },
  { "id": "driver_generic",  // 兜底
    "crates": ["driver"],
    "path_patterns": ["*"],
    "targets": ["全部 board 目标"] }
]
✅ 设计亮点：双重兜底安全网

1. 规则不匹配时回退到全量测试（no_match_fallback_all），永不遗漏
2. Rust 引擎失败时自动降级到 Bash 脚本，CI 不中断
3. crate_path_rules 的 * 通配兜底，通用驱动变更触发全部板卡
